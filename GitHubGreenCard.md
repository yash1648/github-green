## **System Architecture Document: Automated DSA Portfolio Engine** 

## **1. Executive Summary** 

The goal of this system is to maintain a consistent, high-quality development footprint on GitHub while building a professional, human-curated repository of Data Structures and Algorithms (DSA) solutions. The system is designed to run completely autonomously, serverless, and with an emphasis on **high availability and fault tolerance** . It ensures that the GitHub Contribution Graph is updated daily with well-documented, clean solutions that read as if they were written from a human developer's perspective. 

## **2. Comprehensive Architectural Breakdown** 

## **Module 1: Ingestion Engine (The Fail-Safe Scout)** 

The pipeline begins exactly at 01:00 UTC every day, triggered by a serverless cron coordinator via **GitHub Actions** . 

1. **The Gateway Check (API Online?):** The system issues an initial handshake or query to external coding platforms (e.g., LeetCode's GraphQL API). 

2. **The Active Path (Fetch Problem):** If the external platform responds successfully, the engine pulls down the official _Problem of the Day_ . It extracts the problem title, description, constraints, difficulty level, and the starting boilerplate code stub. 

3. **The Passive Path (Local Backlog):** In the event of a network outage, API structural change, or rate-limiting error, a conditional fallback circuit trips. Instead of letting the pipeline crash, the system immediately shifts to a local backlog.json file embedded within the repository containing curated classic DSA questions. 

4. **Unified Output:** Both paths converge to generate a standardized Problem Context JSON object, isolating downstream modules from changes in the ingestion source. 

## **Module 2: Execution Engine (The Cognitive Pipeline)** 

Once the problem context is isolated, it is processed through a dual-stage Large Language Model (LLM) configuration designed to maximize code efficiency and mimic human documentation styles. 

- **Stage 1: LLM Gen 1 (Optimized Code Generator):** This stage consumes the problem description and code stub. It acts strictly as an algorithm expert, 

generating clean, production-grade solution code (e.g., in C++ or Python) optimized for time and space complexity. It runs purely on code correctness. 

- **Stage 2: LLM Gen 2 (The Humanized Dev Journal):** This component takes **both** the original problem description _and_ the code generated by LLM Gen 1 as explicit context. Rather than spitting out a robotic, textbook-like explanation, this model writes a "Developer Journal." It outlines the problem's core trick, the initial thoughts/brute-force ideas, and structural trade-offs. It is explicitly constrained to avoid AI-identifying buzzwords ( _"Furthermore," "Crucial," "Let's dive in"_ ), resulting in a genuine-looking markdown narrative. 

## **Module 3: Repository Management (The Structuring Engine)** 

After the code and the humanized documentation are ready, the workspace manipulator takes over to structure the file system cleanly. 

1. **Directory Isolation:** The engine parses the problem title and automatically constructs a uniform folder layout inside the workspace, for example: 

./LeetCode/YYYY-MM-DD-Problem-Name/ 

2. **Artifact Generation:** Inside this specific folder, two essential artifacts are written: 

   - solution.cpp (or your chosen language file containing the verified code payload). 

   - README.md (the highly professional, humanized breakdown detailing the source URL, thought process, and complexities). 

## **Module 4: Deployment Engine (The Automated Publisher)** 

The final stage bridges the temporary GitHub Actions runner workspace back to your live GitHub profile. 

1. **Git Staging & Tracking:** The system invokes a local Git instance inside the runner to stage the newly added directory (git add .). 

2. **Authentic Commits:** It creates a formatted, clear commit message detailing the problem solved for that calendar day: 

git commit -m "Docs: Auto-solve [LeetCode] Twin Sum - 2026-06-24" 

3. **Secure Push:** Utilizing a securely injected GitHub Token (GITHUB_TOKEN or a Personal Access Token stored in GitHub Secrets), the agent safely pushes the branch upstream to the main branch. 

4. **The End Result:** The push updates the remote repository, triggering GitHub’s 

   - internal tracking metrics to append a green contribution square onto your public profile graph.
